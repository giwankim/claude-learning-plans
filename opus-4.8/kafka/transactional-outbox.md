# A Structured Learning Path: The Transactional Outbox Pattern for Event-Driven Architecture with Apache Kafka

## TL;DR
- The transactional outbox pattern is the canonical fix for the dual-write problem; for your stack (Kotlin/Spring Boot + Aurora MySQL + Kafka on EKS), the strongest path is to **start conceptual** (Richardson's microservices.io + Kleppmann's DDIA Ch. 11), then **specialize to Debezium's transaction-log-tailing approach** (Debezium MySQL connector + Outbox Event Router SMT), because CDC-based outbox preserves per-aggregate ordering and avoids polling-publisher reordering pitfalls.
- The single most relevant real-world reference for your project is the Woowahan **Delivery Service Team** blog post and its WOOWACON 2023 companion talk: they run essentially your exact architecture (single Delivery domain, MySQL → Debezium → Kafka outbox, per-key ordering) at over **1 million Baemin deliveries per day** ("하루 100만 건 이상 생성되는 배민배달… 중계"), and they document a non-obvious production constraint — the Debezium MySQL connector is single-task, so they shard the outbox into multiple tables (delivery-outbox1/2/3) to scale throughput.
- Plan to progress through five phases (0: EDA/dual-write fundamentals → 1: stack-agnostic outbox → 2: minimal Spring+MySQL+Debezium implementation → 3: deep dives on semantics/ordering/ops → 4: Aurora-specific CDC), prioritizing free primary sources (official Debezium docs, Confluent, Spring, AWS) supplemented by two books you should buy.

## Key Findings
- **Two implementation styles, one preferred.** Richardson distinguishes the **Polling Publisher** and **Transaction Log Tailing** approaches. For Aurora MySQL + Debezium, transaction-log-tailing (CDC) is preferred because it reads the binlog rather than polling, gives lower latency, and preserves ordering. The RIDI engineering blog is a candid counterpoint: they chose Polling Publisher and had to do repeated performance tuning (deadlocks, lock waits, latency) — useful evidence of the operational cost of polling.
- **Delivery semantics are fundamentally at-least-once.** The outbox guarantees the event is published at least once; therefore **consumers must be idempotent / deduplicate**. True Kafka exactly-once (EOS) via idempotent + transactional producers only covers the consume-process-produce loop inside Kafka — it does not extend across the Debezium CDC boundary, so you still need consumer-side dedup keyed on event id.
- **Ordering is achieved by aggregate id = Kafka message key.** The Debezium Outbox Event Router maps `aggregateid` to the Kafka message key, so all events for one Delivery land on one partition and stay ordered. This is a hard architectural property of the connector: the Debezium MySQL connector docs state plainly, *"The connector only supports a single task, because MySQL records all of its activities in one sequential binlog. As a result, the connector requires only a single reader to obtain a consistent, ordered view of all of the events."* To scale throughput within that constraint, the Woowahan team partitions the outbox into multiple tables by identifier with one connector each — *"처리량을 높이기 위해 토픽별로 outbox 테이블을 분리하여 만들고… delivery-outbox1, delivery-outbox2, delivery-outbox3과 같이 여러 개의 outbox 테이블을 구성하고, 각 테이블에 커넥터를 연결하여… 같은 키는 같은 테이블에 저장되며, 한 테이블에서는 하나의 커넥터를 사용하기 때문에 같은 키에 대해서는 순서를 보장됩니다."* ("To raise throughput we separated outbox tables per topic… configured multiple outbox tables like delivery-outbox1/2/3 and attached a connector to each… the same key is stored in the same table, and because one table uses one connector, ordering is guaranteed for the same key.")
- **Aurora MySQL has specific CDC prerequisites.** You must enable ROW binlog format and set binlog retention via the stored procedure (not the usual MySQL config). Aurora's default is to purge binlogs aggressively: per AWS, *"The default value of binlog retention hours is NULL. For Aurora MySQL, NULL means binary logs are cleaned up lazily. Aurora MySQL binary logs might remain in the system for a certain period, which is usually not longer than a day"* — too short for a connector that goes down.
- **Spring gives you "outbox-lite" options.** Spring Modulith's Event Publication Registry is effectively a built-in outbox table tied to `ApplicationEventPublisher`, with externalization to Kafka — but ordering is not guaranteed on resubmission. For strict ordering, CDC with Debezium remains the recommended option.

## Details

### Phase 0 — Optional refresher: EDA fundamentals, the dual-write problem, and where outbox fits
Goal: frame the problem and the alternatives. Skim if already fluent.

- **Designing Data-Intensive Applications, Ch. 11 "Stream Processing" — Martin Kleppmann (book, paid, advanced).** The theoretical backbone: log-based message brokers, change data capture, event sourcing, the dual-write race condition, and why "the truth is the log." This is the single best grounding for the distributed-systems theory under CDC. https://www.oreilly.com/library/view/designing-data-intensive-applications/9781491903063/ch11.html
- **microservices.io — Pattern: Transactional outbox (Chris Richardson) (docs/pattern page, free, intermediate).** The canonical pattern statement, including the ordering requirement (E1 before E2) and the note that consumers must be idempotent. https://microservices.io/patterns/data/transactional-outbox.html
- **Confluent Developer — "The Transactional Outbox Pattern" course module (video + text, free, intermediate).** Crisp articulation of the dual-write problem and how pushing the logic into the DB transaction solves it. https://developer.confluent.io/courses/microservices/the-transactional-outbox-pattern/
- **AWS Prescriptive Guidance — Transactional outbox pattern (docs, free, intermediate).** AWS's own framing, useful because you're on AWS. https://docs.aws.amazon.com/prescriptive-guidance/latest/cloud-design-patterns/transactional-outbox.html
- **Decodable — "Revisiting the Outbox Pattern" (Gunnar Morling) (blog, free, advanced).** Situates outbox against alternatives: listen-to-yourself, CDC-only, and Kafka's upcoming 2PC support. On that last point, KIP-939 is explicitly aimed at the dual-write problem: per Confluent, *"KIP-98 introduced exactly-once semantics for data processing within Kafka. KIP-939 brings exactly-once semantics to the next level and enables exactly-once semantics between Kafka and a database… supporting two phase commit and proving an atomic dual-write recipe that can be used to atomically write data to Kafka and a database"* (implemented via the new producer property `transaction.two.phase.commit.enable` and `Producer#initTransactions(keepPreparedTxn)`). Excellent for the "where does it fit vs alternatives" axis. https://www.decodable.co/blog/revisiting-the-outbox-pattern

### Phase 4 — Aurora MySQL–specific CDC concerns
Goal: nail the AWS specifics for your binlog/connector setup.

- **AWS docs — Setting and showing binary log configuration (Amazon Aurora) (docs, free, advanced).** The `mysql.rds_set_configuration('binlog retention hours', N)` stored procedure; default NULL means Aurora purges binlogs lazily ("usually not longer than a day") — too short for a connector that goes down. https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/mysql-stored-proc-configuring.html
- **AWS re:Post — Increase binlog retention in an Aurora MySQL cluster (docs, free, intermediate).** Practical steps: set `binlog_format=ROW` via a custom DB cluster parameter group. Retention max is 168h (7 days) on Aurora MySQL 5.7-compatible; on version 2.11.0+ the maximum is *"2160 (90 days). If the value that you enter for binary log retention hours is greater than 2160, then the value defaults to 2160."* https://repost.aws/knowledge-center/aurora-mysql-increase-binlog-retention
- **AWS docs — Setting up binary log replication for Aurora MySQL (docs, free, advanced).** Binlog enablement and retention-until-applied semantics. https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/AuroraMySQL.Replication.MySQL.SettingUp.html
- **Confluent docs — MySQL CDC Source V2 (Debezium) Connector (docs, free, advanced).** States the recommendation verbatim: *"Confluent recommends setting the binlog retention period to 7 days… If the connector is stopped for longer than the binlog retention period, the required binlog files may be purged. This can cause the connector to fail or require a full snapshot to recover."* https://docs.confluent.io/cloud/current/connectors/cc-mysql-source-cdc-v2-debezium/cc-mysql-source-cdc-v2-debezium.html
- **Practitioner note (community, intermediate):** field blogs document Aurora-specific grants (`LOCK TABLES`, `REPLICATION CLIENT/SLAVE`), the `ROW binlog_format` requirement, and the classic post-downtime failure — verbatim, *"[ERROR] Connector requires binlog file 'mysql-bin.001134', but MySQL only has mysql-bin.001255, mysql-bin.001256"* — for which the fix is `snapshot.mode=when_needed`. Budget retention accordingly. (Sylhare's Debezium troubleshooting blog: https://sylhare.github.io/2023/11/07/Debezium-configuration.html)

## Recommendations
1. **Week 1 (concepts):** Read microservices.io outbox + transaction-log-tailing pages, watch Morling's "Ins and Outs of the Outbox Pattern," and read DDIA Ch. 11. Outcome: you can articulate the dual-write problem and justify CDC-tailing over polling for your stack.
2. **Week 2 (get it running):** Stand up the Debezium MySQL outbox demo locally (docker-compose), then read the Outbox Event Router SMT docs and wire `aggregateid` → message key. Outcome: a Delivery event flows DB → Kafka with per-aggregate ordering.
3. **Week 3 (specialize to your domain):** Study the Woowahan delivery post + WOOWACON talk and the 29CM implementation; decide between a hand-rolled outbox table vs. Spring Modulith's registry. Given your need for strict ordering, lean toward the Debezium CDC outbox, not Modulith resubmission. Outcome: a design doc for the Delivery aggregate outbox.
4. **Week 4 (harden):** Implement consumer-side idempotency/dedup (event-id table), set Avro + schema registry for payloads, add an outbox cleanup job, and instrument connector lag (JMX/Prometheus). Read the RIDI two-parter to anticipate lock/latency issues if you keep any polling path.
5. **Week 5 (Aurora + ops):** Configure `binlog_format=ROW` and set binlog retention to 7 days via the Aurora stored procedure; document the snapshot strategy (initial vs. incremental signaling) and the downtime-vs-retention budget; deploy Connect on EKS via Strimzi.

**Thresholds that change the plan:**
- If throughput on a single Delivery outbox table saturates the single-task Debezium connector, shard the outbox into N tables by aggregate id (the Woowahan approach) before reaching for more connectors.
- If you do NOT need cross-aggregate ordering and can tolerate Modulith's resubmission reordering, Spring Modulith's registry may remove the need for Debezium entirely — simpler ops, fewer moving parts.
- If KIP-939 (Kafka 2PC, via `transaction.two.phase.commit.enable`) ships and matures, re-evaluate whether the outbox is still needed for your write path (per Morling's "Revisiting" post).

## Caveats
- **At-least-once is the ceiling for the outbox itself.** Any claim of "exactly-once" end-to-end is misleading: Kafka EOS covers in-Kafka consume-process-produce, not the DB→Debezium→consumer chain. Design consumers to be idempotent. (KIP-939 aims to close exactly this gap but is still emerging — treat it as forward-looking, not production-ready, until you verify its release status in your Kafka version.)
- **The single-task Debezium MySQL connector is a hard constraint**, stated in the connector docs and confirmed in production by the Woowahan team; plan your outbox sharding strategy early rather than discovering it under load.
- **Source-quality notes:** microservices.io, Debezium docs, Confluent, Spring, and AWS are authoritative primary sources. Medium/Velog/DEV posts (including the Korean blogs) are practitioner experience reports — high signal for your domain but verify config specifics against official docs, since connector property names and Debezium versions drift. The Woowahan post's "order-cancel" discussion is about data-consistency (a canceled delivery still being processed if the event fails to publish), not a literal cancel-before-create race; the out-of-order example they actually give is dispatch-complete (배차완료) vs. pickup-prep request (픽업준비요청). The WOOWACON talk's speaker (임준수, Delivery Service Team) and high-level scope are verified from the official listing, but its minute-by-minute content was not independently transcribed.
- **Korean resources are explicitly included and labeled** per your bilingual preference; they are among the most project-relevant items here and are not merely supplementary.
- This path assumes Debezium 2.x/Kafka Connect; verify exact SMT class names, snapshot signaling, and Aurora version-dependent retention limits against the versions you deploy.
