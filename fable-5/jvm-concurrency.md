# A From-Fundamentals Learning Plan for JVM Concurrency (2026 Edition)

## TL;DR
- **Start with the classical core** (Java Memory Model, threads, `java.util.concurrent`, locks, atomics, lock-free structures) using *Java Concurrency in Practice* + Aleksey Shipilëv's JMM writings + the Rice Coursera specialization, then layer on the modern JVM (virtual threads/Loom, structured concurrency, scoped values) and Kotlin coroutines — this order matters because coroutines and virtual threads are only fully intelligible once you own happens-before, safe publication, and executor/blocking semantics.
- **Verified 2026 state**: Java 25 is the current LTS; **scoped values are FINAL (JEP 506)** and **virtual-thread pinning on `synchronized` is essentially fixed (JEP 491, delivered in JDK 24)**, but **structured concurrency is still preview** (JEP 505 in JDK 25, JEP 525 in JDK 26, JEP 533 targeted for JDK 27) — so treat `StructuredTaskScope` as learn-now/ship-carefully. kotlinx.coroutines is at 1.11.x (Kotlin 2.2.x); Spring Boot enables virtual threads with one property (`spring.threads.virtual.enabled=true`) since 3.2.
- **A phased ~12–18 month roadmap**: Phase 1 (months 1–3) foundations; Phase 2 (months 3–9) `j.u.c` mastery, lock-free, coroutines internals, Loom internals, testing with Lincheck/jcstress/JMH; Phase 3 (months 9–18+) research-grade theory (linearizability, memory models), production tuning tied to your Kafka/Aurora/EKS stack, and a mini structured-concurrency library. Concrete hands-on projects anchor every phase.

## Key Findings

**1. The classical core is not "legacy" — it is the substrate.** Virtual threads are still threads; coroutine dispatchers still wrap executors; `ConcurrentHashMap`, `CompletableFuture`, and the JMM's happens-before are as load-bearing in 2026 as in 2006. Your pure-math background is a genuine advantage here: the field has a rigorous formal core (linearizability, memory-consistency models, wait-free hierarchies) that most engineers skip and that pays off directly when reasoning about lock-free code and weak memory behaviors.

**2. The 2026 landscape shifted the tradeoffs, not the fundamentals.** JEP 491 (JDK 24) removed the biggest virtual-thread footgun — pinning inside `synchronized` — which had caused a well-documented Netflix production incident on Java 21. Critically, per JEP 491 the fix **requires no code changes**: "Existing applications with synchronized blocks automatically benefit from the fix when they upgrade to Java 24 (or Java 25 LTS, which inherits the fix)." This changes *advice* ("you can use `synchronized` again") but not *understanding* — you still must know why pinning happened to reason about the remaining native/JNI cases.

**3. Virtual threads vs. coroutines is a real, ongoing decision, and Korean engineering blogs have unusually strong primary material on it.** Woowahan and Kakao Pay both published head-to-head benchmarks and decision frameworks in Korean — directly usable given you work bilingually.

**4. Your stack gives you natural lab equipment.** Kafka consumer threading, Spring `@KafkaListener` concurrency, HikariCP pool sizing (Little's Law + the Universal Scalability Law), and Spring Boot's virtual-thread switch are all concrete, high-value places to make abstract concurrency concrete.

## Details

### Verified current-state facts (as of July 2026)
- **Java LTS**: Java 25 is the current LTS. **Scoped Values** are finalized as a standard API in JDK 25 (JEP 506) — one behavior change vs. preview: `orElse` no longer accepts null. **Structured Concurrency** remains preview: JEP 505 (JDK 25, 5th preview — a major API rework that decoupled `StructuredTaskScope` from the join policy via a `Joiner` and moved to static `open()` factory methods, "composition over inheritance"), JEP 525 (JDK 26, 6th preview — `join()` now returns the result list directly), and JEP 533 (targeted JDK 27, 7th preview — planned to switch standard joiners to throw `ExecutionException` instead of `FailedException`).
- **Pinning**: JEP 491 "Synchronize Virtual Threads without Pinning" was delivered in JDK 24. Monitor ownership is now tracked per virtual thread rather than per carrier; the `-Djdk.tracePinnedThreads` system property was removed; use the `jdk.VirtualThreadPinned` JFR event (enabled by default, ~20 ms threshold). Remaining pinning cases: native/JNI calls and Foreign Function & Memory API callbacks, plus rare class-initialization edge cases.
- **kotlinx.coroutines**: 1.11.x, the companion release line for Kotlin 2.2.x.
- **Spring Boot**: `spring.threads.virtual.enabled=true` (available since 3.2) makes Tomcat/Jetty use a virtual-thread-per-request executor and auto-configures `SimpleAsyncTaskExecutor` (with `setVirtualThreads(true)`) as the default `@Async` executor; it also affects `@Scheduled` tasks, Kafka/RabbitMQ listeners, Spring Data Redis, and blocking WebFlux calls. Watch two pitfalls: ThreadLocal/MDC memory and inheritance behavior (always `MDC.clear()` in a `finally`), and connection-pool sizing (below).

### Phase 1 — Foundations (Months 1–3, ~6–8 hrs/week)
**Topics**: thread lifecycle and states; `Runnable`/`Callable`; `synchronized`, monitors, `wait`/`notify`; the Java Memory Model — happens-before, `volatile`, final-field semantics, safe publication, data races, "out of thin air" values; the fundamentals of `java.util.concurrent`: `ExecutorService`, `Future`, thread pools.

**Resources**:

- **Book (reference)**: *Effective Java* (3rd ed.), Bloch — the concurrency chapter.
- **Primary sources**: JLS Chapter 17 (Threads and Locks / the JMM); Aleksey Shipilëv — "Java Memory Model Pragmatics" (shipilev.net/blog/2014/jmm-pragmatics/), "Safe Publication and Safe Initialization" (2014), "Close Encounters of the JMM Kind" (2016); Doug Lea's "JSR-133 Cookbook for Compiler Writers" (gee.cs.oswego.edu/dl/jmm/cookbook.html).
- **Course**: Rice University "Parallel, Concurrent, and Distributed Programming in Java" specialization on Coursera (Prof. Vivek Sarkar). The Concurrency course explicitly covers threads, locks, atomics, concurrent collections, and their theoretical foundations (linearizability, progress guarantees, memory models).
- **Korean**: Devsisters "바삭한 신입들의 동시성 이야기 - Kotlin 편" (tech.devsisters.com/posts/crunchy-concurrency-kotlin/) for the CPS/continuation intuition in your working language.

**Projects**:
- P1a: Build a fixed-size thread pool from scratch (blocking queue + worker threads + graceful shutdown), then compare against `ThreadPoolExecutor`.
- P1b: Write JMM "litmus tests" by hand (store buffering; safe vs. unsafe publication of a config object) and predict legal outcomes *before* running them.

### Phase 2 — Deepening & modern JVM (Months 3–9, ~6–10 hrs/week)
**Topics**: full `java.util.concurrent` — `CompletableFuture` composition, `CountDownLatch`, `Semaphore`, `CyclicBarrier`, `Phaser`, `ReentrantLock`/`ReadWriteLock`/`StampedLock`, `ConcurrentHashMap`, `BlockingQueue` family; atomics and CAS, the ABA problem, `LongAdder`; `ForkJoinPool` and work-stealing; lock-free/wait-free structures (Treiber stack, Michael-Scott queue); **virtual threads & Loom internals** (continuations, carrier mounting/unmounting, M:N scheduling, pinning, monitoring); **structured concurrency** (`StructuredTaskScope` + `Joiner`); **scoped values vs. `ThreadLocal`**; **Kotlin coroutines internals** (suspension, continuation-passing style, dispatchers, `Job`/structured concurrency, channels, flows, `SharedFlow`/`StateFlow`, exception handling); a brief reactive context (Project Reactor/RxJava and why virtual threads reduce the need for them); **testing** (jcstress, Lincheck, JMH).

### Phase 3 — Mastery, theory & production (Months 9–18+, ~4–8 hrs/week)

**Resources**:
- **Papers**: the three foundational papers above (all directly relevant to lock-free reasoning and linearization points); the Lincheck CAV'23 paper; optionally "LitmusKt: Concurrency Stress Testing for Kotlin" (arXiv:2501.07472) for memory-model testing on Kotlin.
- **Books**: revisit *Art of Multiprocessor Programming* chapters on consensus, universality, and transactional memory now that you have the practice under your belt.
- **Stack-specific**: HikariCP "About Pool Sizing" wiki (Brett Wooldridge) — the heuristic verbatim: "for optimal throughput the number of active connections should be somewhere near ((core_count * 2) + effective_spindle_count). Core count should not include HT threads... Effective spindle count is zero if the active data set is fully cached." The same page documents an Oracle Real-World Performance demo where "reducing the connection pool size alone, in the absence of any other change, decreased the response times of the application from ~100ms to ~2ms — over 50x improvement," and recommends "a small pool of a few dozen connections at most." Pair with Vlad Mihalcea's "The best way to determine the optimal connection pool size" (Universal Scalability Law + FlexyPool) and apply Little's Law (L = λW) as directional guidance for average concurrency. Spring Kafka `@KafkaListener` docs and Piotr Minkowski's "Concurrency with Kafka and Spring Boot" — key rule: `concurrency` ≤ partition count (extra threads sit idle), thread-per-partition preserves per-partition ordering, and offloading to an app-level `ExecutorService` regains parallelism but *loses* single-partition ordering.
- **Observability**: use `jcmd <pid> Thread.dump_to_file -format=json` to see virtual-thread and scope hierarchies; async-profiler for wall-clock/CPU flame graphs; JFR `jdk.VirtualThreadPinned` events to catch remaining native-pinning.

### Honest tradeoffs & decision frameworks
- **When NOT to use virtual threads**: CPU-bound work (use platform threads / parallel streams / ForkJoin); code with heavy native/JNI on the hot path (still pins post-JEP 491); and **never pool virtual threads** — they are cheap to create, so use a `Semaphore` to bound concurrency against a downstream dependency instead of a fixed pool. Virtual threads improve *throughput*, not *latency*.
- **Coroutines vs. virtual threads**: coroutines win for structured concurrency, fine-grained cancellation, flows/event streams, high-concurrency fan-out/merge, and pre-JDK-21 runtimes; they carry the "function color" (`suspend`) propagation cost and demand strict non-blocking discipline (a stray blocking call can stall a dispatcher). Virtual threads win for thread-per-request blocking code, reuse of existing blocking libraries, and Java-ecosystem generality. They are complementary — a coroutine dispatcher can run atop a virtual-thread executor, and the Woowahan/Kakao Pay posts frame them as coexisting rather than competing.
- **Lock-free complexity cost**: lock-free code is dramatically harder to get right (ABA, safe memory reclamation, subtle linearization points) and *must* be verified with Lincheck/jcstress; prefer `j.u.c` structures unless profiling proves a specific bottleneck. This is exactly where your formal background earns its keep — but it is also where confidence without model-checking is most dangerous.

## Recommendations
1. **Months 1–3 (foundations)**: Read *JCIP* cover-to-cover alongside Shipilëv's "JMM Pragmatics"; complete the Rice Concurrency course; ship P1a–P1c. **Advance when**: you can explain happens-before from memory and justify every `volatile`/`final` in a safe-publication example without notes.
2. **Months 3–9 (deepening)**: Work *Art of Multiprocessor Programming* (through lock-free structures) and *Kotlin Coroutines: Deep Dive* in parallel; make Lincheck + JMH standing habits; ship P2a–P2e. **Advance when**: a Lincheck test of yours catches a real interleaving bug, and your load test cleanly reproduces + resolves the JDK 21→24 pinning story.
3. **Months 9–18+ (mastery)**: Read Lamport (1978) and Herlihy–Wing (1990); build the mini structured-concurrency library (P3a); run the Kafka + HikariCP production studies (P3b–P3c). **Advance when**: you can derive a defensible connection-pool size from Little's Law + core count and defend a coroutines-vs-virtual-threads choice for a specific service with your own data.
4. **Thresholds that change the plan**: if your team standardizes on virtual threads across services, front-load Phase 2's Loom internals and observability (P2d, P3d) ahead of lock-free theory. If your team moves toward heavy reactive/streaming, prioritize flows/channels and Project Reactor. If you stay predominantly Kotlin + Spring MVC with blocking I/O, treat virtual threads (not WebFlux) as your default scaling lever and invest most in coroutines for the genuinely concurrent fan-out paths.

## Caveats
- Structured concurrency is still a **preview** feature through at least JDK 27; its API changed materially between JDK 21 and 25 (the `Joiner` rework) and will change again in 27 (exception types). Learn the concept and prototype now, but gate production use behind `--enable-preview` awareness and expect churn.
- *Java Concurrency in Practice* (2006) is authoritative on fundamentals but silent on everything after Java 5 (`CompletableFuture`, Loom, modern GCs); always cross-check API specifics against current JDK docs.
- Benchmark figures cited here are **hardware- and workload-specific single-run results**, not universal laws. Woowahan's "37% faster" was measured at 510 vusers on their specific I/O-bound test; various "~2x" Spring Boot virtual-thread numbers and the HikariCP "50x" figure come from particular demos. Treat all of them as directional and re-measure on your own EKS/Aurora setup.
- The Naver D2 URL was verified by strong cross-reference, but d2.naver.com blocks automated fetching — confirm it loads in a browser. Likewise, a `tech.kakao.com/posts/608` virtual-thread post is cited by third parties but was not directly opened during research; verify before relying on it.
