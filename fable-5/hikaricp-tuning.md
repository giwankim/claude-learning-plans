# HikariCP Connection Pool Tuning — Learning Guide & Study Plan
*For a senior backend engineer (Kotlin / Spring Boot / Aurora MySQL / Kafka / EKS)*

## TL;DR
- HikariCP is a mature, mostly-stable library: the current release is **7.1.0** (a "Java 11+" Maven artifact), all of the 5.x/6.x/7.x line requires **Java 11+**, and the core tuning wisdom ("a small, fixed-size pool saturated with waiting threads beats a large one") has not changed since the "About Pool Sizing" wiki was written — so your study time should go into *empirical* sizing, Aurora failover behavior, and observability, not into chasing new APIs.
- The single most important mental model to internalize is the deadlock formula `pool_size = Tn × (Cm − 1) + 1` and the Woowahan incident that proves it: a `@GeneratedValue(AUTO/SEQUENCE)` insert needs **2 connections per thread**, so a pool smaller than the concurrent thread count deadlocks under load — this is the centerpiece lab.
- The Aurora-specific traps (synchronized `maxLifetime` "mass extinction" on the reader endpoint, `maxLifetime` vs `wait_timeout` causing "communications link failure," and DNS-based failover being slow without the AWS Advanced JDBC Wrapper) are where your stack differs most from generic guides, so Phase 3 is Aurora-heavy.

## Key Findings

1. **The canonical guidance is stable and primary-source-rich.** HikariCP's own wiki ("About Pool Sizing," "Down the Rabbit Hole," "Welcome to the Jungle," "Rapid Recovery") plus the README config reference remain the authoritative texts. Nothing in 6.x/7.x invalidates them.
2. **Version reality check (correction to a common myth):** HikariCP did **not** replace `synchronized` with `ReentrantLock` in 6.0.0. That community PR (#2055 "Add support for Virtual Threads") was **reverted**, and Issue #2293 shows users still requesting it. Virtual-thread friendliness instead arrived as `ConcurrentBag` yield-spin tuning in the 7.x line (CHANGES 7.1.0: *"merged #2402 avoid virtual-thread yield spin in ConcurrentBag"*), and JDK 24's JEP 491 largely fixed synchronized-pinning at the JVM level anyway. HikariCP 6.0.0 (Sep 23, 2024) *did* add JDBC 4.3 request boundaries (*"merged #2126 added Support For beginRequest and endRequest"*).
3. **"Smaller is faster" is empirically reproducible.** The formula `connections = (core_count × 2) + effective_spindle_count` is a *starting point*; the Oracle Real-World Performance demo shows response time dropping ~100ms → ~2ms (50x) from shrinking a pool alone. Your lab reproduces this with a maximumPoolSize sweep.
4. **Multi-replica Kubernetes math is the real-world constraint.** Per-pod pool size must be computed against Aurora `max_connections` across all replicas with headroom, e.g. `per_pod = (max_connections × 0.8) / num_pods`.
5. **Aurora needs special handling** that generic Postgres/MySQL guides miss: reader-endpoint DNS load-balancing pitfalls, the AWS Advanced JDBC Wrapper for fast failover, and RDS Proxy tradeoffs. Per AWS's "Unlock Amazon Aurora's Advanced Features" blog, "When Aurora fails over in seconds, standard JDBC drivers can take up to a minute to reconnect because of DNS propagation delays." RDS Proxy is most valuable for connection-storm workloads — AWS's product page cites CloudHealth by VMware (Peter Fein) reducing "database connections by up to 90 percent during peak hours, insulating the database from high connection counts."
6. **Korean-language resources are excellent and directly on-point** — the Woowahan two-parter and Carrey's blog are the best deadlock write-ups in any language; Netmarble and Yun/Kakao-style engineering blogs add game-server and latency angles.

- **Oracle Real-World Performance video** (linked from that wiki page) — https://www.youtube.com/watch?v=_C77sBcAtSQ — ~10-min demo where reducing pool size alone cut response time ~100ms → ~2ms (50x). **Why:** the single most persuasive artifact for "smaller is faster."
- **Wiki — "Down the Rabbit Hole"** — https://github.com/brettwooldridge/HikariCP/wiki/Down-the-Rabbit-Hole — Internals: `FastList` (replaces ArrayList, kills range-checks, tail-to-head removal scan), `ConcurrentBag` (lock-free, ThreadLocal caching, queue-stealing, direct hand-off), and bytecode/JIT-inline optimizations. **Why:** matches your JVM-concurrency depth and explains virtual-thread interactions.
- **Wiki — "Welcome to the Jungle"** — https://github.com/brettwooldridge/HikariCP/wiki/Welcome-to-the-Jungle — Spike-demand handling and connection elision: in a test (150ms connect time, 50 simultaneous requests) HikariCP served all 50 while creating only 1 extra connection. **Why:** explains why fixed pools + elision beat dynamic pools.
- **Wiki — "Rapid Recovery"** — https://github.com/brettwooldridge/HikariCP/wiki/Rapid-Recovery — Failover/aliveness: once a connection is borrowed, HikariCP can't recover it from a network partition due to unacknowledged TCP; recommends driver-level socketTimeout ≥ 2–3x longest transaction or 30s, and warns about JVM/OS DNS caching. **Why:** directly relevant to Aurora failover.
- **DeepWiki mirror** — https://deepwiki.com/brettwooldridge/HikariCP/4.2-pool-sizing-and-performance-tuning — A searchable, cross-referenced restatement of the wiki with source-file citations. **Why:** convenient for quick lookup.

- **OneUptime — Tune HikariCP for maximum throughput** (Jan 2026) — https://oneuptime.com/blog/post/2026-01-25-tune-hikaricp-maximum-throughput-spring-boot/view — Clarifies the effective-spindle-count nuance (no special SSD value), fail-fast philosophy, validation-timeout < connection-timeout.
- **DBAGlobe — Critical connection-pool parameters** (Mar 2026) — https://www.dbaglobe.com/2026/03/critical-connection-pool-parameters-and.html — Source-code-verified defaults (maxPoolSize 10, connectionTimeout 30000, idleTimeout 600000, maxLifetime 1800000, minIdle=-1 ⇒ equals max) and the maxLifetime > wait_timeout anti-pattern. **Why:** precise default reference.
- **Spring Boot property names**: all under `spring.datasource.hikari.*` (e.g. `maximum-pool-size`, `minimum-idle`, `connection-timeout`, `idle-timeout`, `max-lifetime`, `keepalive-time`, `leak-detection-threshold`, `validation-timeout`, `pool-name`). Identical keys across Spring Boot 2.x and 3.x. Note: since Spring Boot 2.3 the default max pool size is dynamic: `max(10, availableProcessors × 2 + 1)`.
- **Vlad Mihalcea — optimal pool size / FlexyPool** — https://vladmihalcea.com/optimal-connection-pool-size/ and https://vladmihalcea.com/connection-pool-sizing-with-flexy-pool/ — Universal Scalability Law framing; FlexyPool's `IncrementPoolOnTimeout` strategy empirically discovers the equilibrium size (his example: 64 transfers run on just 4 connections; lowering pool from a large size increases contention/retries). **Why:** rigorous methodology and a tool to automate sizing experiments.

#### C. Aurora MySQL specifics
- **Amazon Aurora MySQL DBA's Handbook — Connection Management** — https://docs.aws.amazon.com/whitepapers/latest/amazon-aurora-mysql-db-admin-handbook/amazon-aurora-mysql-db-admin-handbook.html (PDF: https://d1.awsstatic.com/whitepapers/RDS/amazon-aurora-mysql-database-administrator-handbook.pdf) — Smart drivers, DNS caching, connection validation with `@@innodb_read_only` to detect reader vs writer, recycle connections periodically (~every 15 min), tune down `wait_timeout`/`interactive_timeout` from the 28,800s (8-hour) default, and Aurora's server-side thread-pool/multiplexing model. **Why:** the authoritative Aurora connection doc.
- **AWS Advanced JDBC Wrapper** — https://github.com/aws/aws-advanced-jdbc-wrapper — Wraps the MySQL/Postgres driver, maintains a real-time topology cache to bypass DNS delays; plugins `failover`/`failover2`, `efm2` (Enhanced Failure Monitoring), `auroraConnectionTracker`, read/write splitting. **Critical HikariCP integration detail:** you must set `exceptionOverrideClassName=software.amazon.jdbc.util.HikariCPSQLException` or connections get closed immediately after failover. Current major version is 3.x (clusterId became mandatory for multiple clusters in 3.0.0); supports in-region + cross-region planned failover of Aurora Global Databases as of 3.0.0.
- **AWS blog — Unlock Aurora features with the JDBC Wrapper** — https://aws.amazon.com/blogs/database/unlock-amazon-auroras-advanced-features-with-standard-jdbc-driver-using-aws-advanced-jdbc-wrapper/ ; **Demystifying the plugins** — https://aws.amazon.com/blogs/database/demystifying-the-aws-advanced-jdbc-wrapper-plugins/ ; **Introducing the Wrapper** — https://aws.amazon.com/blogs/database/introducing-the-advanced-jdbc-wrapper-driver-for-amazon-aurora/ — Failover v2 + Aurora Initial Connection Strategy. Dave Cramer's introductory post notes DNS "can take up to 30 seconds to resolve... the driver can switch over the connection in around 6 seconds"; the "Demystifying" post reports "With the Failover v2 plugin enabled the client averaged a reconnection time of 7.8 seconds across all tests, whereas the test without the Failover v2 plugin averaged 15 seconds." Demo repo: https://github.com/aws-samples/sample-aws-advanced-jdbc-wrapper-demo.
- **HikariCP issue #1247 — "mass extinction" on Aurora reader endpoint** — https://github.com/brettwooldridge/HikariCP/issues/1247 — Reporter (HikariCP 3.2.0, Aurora MySQL 5.6.10a, pool size 10) observed connection generations recycling "'exactly' 30 minutes" apart, so Aurora's reader endpoint reassigned whole pools to one replica while others sat idle. HikariCP's mitigation, per CHANGES: *"change default maxLifetime variance from 2.5% to 25% to further avoid mass connection die-off dips."* **Why:** the canonical Aurora-reader-load-balancing gotcha.
- **maxLifetime vs wait_timeout** — set HikariCP `maxLifetime` at least ~30s below MySQL `wait_timeout` (and below any load-balancer/NLB idle timeout, often 60–350s on RDS), or you get `CommunicationsException: Communications link failure`. On Aurora the LB timeout is usually the binding constraint, not the 8-hour `wait_timeout`. Refs: https://www.jusdb.com/blog/mysql-timeout-variables-complete-guide-for-database-professionals and https://joshefin.xyz/mariadb-hikaricp-and-connection-lifetime/.
- **RDS Proxy vs client-side pooling** — https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/rds-proxy.html — RDS Proxy multiplexes many client connections into few DB connections (transaction-level reuse), and improves failover resilience. AWS's Database Blog "Improving application availability with Amazon RDS Proxy" measured failover of "3.1 seconds, as opposed to 24 seconds when connecting directly to the database with a MariaDB driver... an 87% improvement," and states RDS Proxy "reduces client recovery time after failover by up to 79% for Amazon Aurora MySQL and by up to 32% for Amazon RDS for MySQL" (the generic "up to 66%" is AWS's cross-engine Multi-AZ figure). Tradeoffs: adds a network hop/latency, session pinning defeats multiplexing, and it adds little value if you already have efficient long-lived client-side pools. **Rule of thumb:** keep HikariCP for steady long-lived services; add RDS Proxy for bursty/serverless (Lambda) or connection-storm workloads.

#### D. Monitoring & observability
- **Micrometer HikariCP metrics** (auto-configured by Spring Boot Actuator + Micrometer): gauges `hikaricp.connections`, `.active`, `.idle`, `.pending`, `.max`, `.min`; timers `hikaricp.connections.acquire`, `.usage`, `.creation`; counter `hikaricp.connections.timeout`. Source: HikariCP `MicrometerMetricsTracker` — https://github.com/brettwooldridge/HikariCP/blob/dev/src/main/java/com/zaxxer/hikari/metrics/micrometer/MicrometerMetricsTracker.java. Enable via `management.endpoints.web.exposure.include=health,metrics,prometheus`.
- **How to interpret:** `pending > 0` ⇒ pool undersized for that load; `active` pinned at `max` ⇒ saturation; rising `.acquire` time with low `.usage` ⇒ contention; any `.timeout` increase ⇒ exhaustion (investigate immediately). Per the wiki axiom, a small pool with some pending threads while `.usage`/`.acquire` stay in single-digit ms is *healthy*, not broken.
- **Interpretation reference** — https://medium.com/@ashah.dev.in/hikaricp-prometheus-metrics-explained-c16c960871ef — thresholds like utilization >80% for 5 min, pending >30s, acquire >100ms.
- **FlexyPool** — https://github.com/vladmihalcea/flexy-pool — Adds histograms (concurrent connections, acquisition time, connection lease time, overflow, retries) and reactive resize strategies on top of HikariCP.

#### E. JVM concurrency angle
- **Deadlock formula** `pool_size = Tn × (Cm − 1) + 1` (from "About Pool Sizing"): Tn = max threads, Cm = max simultaneous connections held by one thread. Nested transactions and `@GeneratedValue` ID generation push Cm to 2. Note the wiki's aside that JTA/a real transaction manager can reduce Cm by handing the same connection back to a thread already in a transaction.
- **Virtual threads (Java 21+)** — HikariCP issue #2151 (https://github.com/brettwooldridge/HikariCP/issues/2151) documents surprising throughput swings under virtual threads; root cause historically was `synchronized` blocks pinning virtual threads to carriers. Reality: PR #2055 (synchronized→ReentrantLock) was reverted; 7.x instead tunes `ConcurrentBag` yield-spin (7.0.1/7.0.2 reduced yield frequency; 7.1.0 #2402 avoids the virtual-thread yield spin), and JDK 24 JEP 491 fixes pinning at the JVM level. **Guidance:** with virtual threads, the pool is still the concurrency limiter — thousands of virtual threads still contend for N connections, so size the pool to DB capacity, not thread count.
- **Tomcat threads vs pool:** default Tomcat max is 200 threads; do not size the pool to match — most threads aren't hitting the DB simultaneously. Measure `hikaricp.connections.active` under load.

## Recommendations
1. **Start now with Labs (a)→(c)**; they deliver 80% of the practical value and directly build the dashboard and deadlock intuition you'll reuse everywhere. Do not skip building the Grafana dashboard first — every later lab is read through it.
2. **Treat the formula as a hypothesis, not an answer.** Use the empirical sweep (Lab b) and FlexyPool to find *your* equilibrium; the wiki's `(cores×2)+1` is where you *start* the sweep.
3. **For Aurora, adopt the AWS Advanced JDBC Wrapper** (`failover2`+`efm2`, `exceptionOverrideClassName` set) unless you have a specific reason not to; it converts DNS-bound reconnections (up to ~a minute) into ~single-digit-second reconnections (AWS-measured ~7.8s with Failover v2). Add RDS Proxy only for bursty/serverless or connection-storm patterns, accepting the added hop.
4. **Always set `maxLifetime` below the tightest infrastructure timeout** (LB idle timeout on RDS, not the 8-hour `wait_timeout`), and rely on the 25% jitter to avoid reader-endpoint mass extinction.
5. **Thresholds that change the plan:** if `hikaricp.connections.pending` is chronically >0 with `usage` p99 low and DB CPU not saturated → increase pool. If `pending` >0 but DB CPU/IO is saturated → the pool is *not* the problem; fix queries / add read replicas / shorten transactions instead. If you see connection timeouts with low active usage → hunt a leak (Lab d), don't raise the pool.

## Caveats
- Several practical guides cited (OneUptime, CodeWiz, DBAGlobe, Medium posts) are secondary sources; treat their config snippets as illustrative and defer to the HikariCP wiki/README/CHANGES and the AWS whitepaper for authority.
- The AWS Advanced JDBC Wrapper is evolving quickly (3.x introduced a mandatory `clusterId` for multi-cluster setups); verify the exact current version and plugin names against the GitHub repo before adopting, and note reported edge-case bugs on Aurora MySQL Blue/Green (`enableGreenNodeReplacement`, issues #1059/#1097).
- Failover timing numbers (~6s legacy wrapper, ~7.8s Failover v2, up to ~a minute for plain DNS, 79%/87% RDS Proxy improvements) come from AWS's own tests and will vary with instance class, workload, and driver version — measure in your environment (Lab f).
- The Java 11 baseline for 5.x/6.x/7.x is documented in the README and confirmed by bytecode class-file versions, not stated explicitly in the CHANGES file itself.
- Virtual-thread performance with HikariCP is still an area of active change; benchmark on your target JDK rather than trusting blog numbers.
