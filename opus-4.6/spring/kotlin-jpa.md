# Production guide to Spring Data JPA with Kotlin

**Kotlin and JPA were designed with conflicting philosophies — Kotlin favors immutability, null safety, and final classes, while JPA demands mutability, no-arg constructors, and subclassable entities.** Making them work together requires understanding exactly where these models collide and applying precise compiler plugin configuration, entity design patterns, and performance practices. This guide covers every major decision point for Spring Boot 3.x with Hibernate 6.x on Aurora MySQL/PostgreSQL, explaining the internal mechanics behind each recommendation.

---

## 1. Entity design: regular classes, never data classes

The single most common mistake in Kotlin+JPA codebases is using `data class` for entities. This breaks three JPA invariants simultaneously. First, Kotlin data classes are **final and cannot be marked `open`** — Hibernate uses ByteBuddy to generate proxy subclasses for lazy-loaded `@ManyToOne` and `@OneToOne` associations, and final classes silently prevent proxy creation, degrading all lazy associations to eager fetching. Second, auto-generated `equals()`/`hashCode()` includes every constructor property, which triggers lazy collection loading when collections are included and breaks `HashSet`/`HashMap` behavior when the `id` changes from `null` to an assigned value after `persist()`. Third, auto-generated `toString()` references all properties including lazy collections, causing `LazyInitializationException` from innocent log statements.

Use a regular `class` with manual `equals()`/`hashCode()`/`toString()`:

```kotlin
@Entity
@Table(name = "posts")
class Post(
    @Column(nullable = false)
    var title: String
) {
    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "post_seq")
    @SequenceGenerator(name = "post_seq", sequenceName = "post_sequence", allocationSize = 50)
    var id: Long? = null

    @OneToMany(mappedBy = "post", cascade = [CascadeType.ALL], orphanRemoval = true)
    var comments: MutableList<Comment> = mutableListOf()

    fun addComment(comment: Comment) {
        comments.add(comment)
        comment.post = this
    }

    fun removeComment(comment: Comment) {
        comments.remove(comment)
        comment.post = null
    }

    override fun equals(other: Any?): Boolean {
        if (this === other) return true
        if (other !is Post) return false
        return id != null && id == other.id
    }

    override fun hashCode(): Int = javaClass.hashCode()

    override fun toString(): String = "Post(id=$id, title=$title)"
}
```

### The four compiler plugins and their exact roles

This is where most teams get the configuration wrong. Four plugins exist, but you need a specific combination of them:

**`kotlin("plugin.spring")`** wraps `kotlin-allopen` with presets for `@Component`, `@Transactional`, `@Cacheable`, and their meta-annotations (`@Service`, `@Repository`, `@Configuration`). It removes the `ACC_FINAL` bytecode flag from annotated classes and all their members, allowing Spring's CGLIB proxies to work.

**`kotlin("plugin.jpa")`** wraps `kotlin-noarg` with presets for `@Entity`, `@Embeddable`, and `@MappedSuperclass`. It generates a **synthetic no-arg constructor** (marked `ACC_SYNTHETIC` in bytecode) that is invisible to Kotlin/Java source code but accessible via reflection — exactly how Hibernate instantiates entities.

**The critical gap**: `plugin.jpa` only adds no-arg constructors. It does **not** open entity classes. Without explicitly configuring `allOpen` for JPA annotations, your entities remain final, proxy generation silently fails, and every `@ManyToOne`/`@OneToOne` association loads eagerly. This is tracked as **KT-79389** and will eventually be fixed upstream.

The correct Gradle Kotlin DSL configuration:

```kotlin
plugins {
    kotlin("jvm") version "2.1.0"
    kotlin("plugin.spring") version "2.1.0"
    kotlin("plugin.jpa") version "2.1.0"
}

// REQUIRED: plugin.jpa does NOT open classes — you must do this manually
allOpen {
    annotation("jakarta.persistence.Entity")
    annotation("jakarta.persistence.MappedSuperclass")
    annotation("jakarta.persistence.Embeddable")
}
```

### Nullability and the no-arg constructor

The synthetic no-arg constructor generated by `kotlin-noarg` does **not** execute `init` blocks or property initializers (unless `invokeInitializers = true` is set). Hibernate calls this constructor via reflection, then populates fields via reflection. This means a non-nullable `val name: String` will temporarily hold `null` at the JVM level, violating Kotlin's type system contract. Hibernate doesn't care — it overwrites values immediately — but the implication matters: **do not rely on `init` blocks for validation in JPA entities**, and be aware that non-nullable types provide compile-time safety for your code but not a runtime guarantee from the persistence layer.

For non-nullable columns, three patterns work:

- **Constructor parameters** (`var name: String`) — enforces non-null at construction time, the most Kotlin-idiomatic approach
- **`lateinit var`** (`lateinit var name: String`) — useful for non-primitive types that don't need defaults, but restricted to `var` and non-primitives
- **Nullable with default** (`var name: String? = null`) — safest for columns that might be null in the database

For the entity ID, **always use `var id: Long? = null`**. The `null` value cleanly communicates "not yet persisted" and correctly triggers Spring Data's `isNew()` detection, which calls `persist()` for new entities. Using `id: Long = 0` causes Spring Data to call `merge()` instead (issuing a SELECT before INSERT), unless you implement the `Persistable<Long>` interface.

### equals()/hashCode() with the constant-hashCode pattern

For entities without a natural business key, use the identifier-based approach with a **constant `hashCode()`** — returning `javaClass.hashCode()` ensures the hash bucket never changes when the `id` transitions from `null` to assigned. In `equals()`, two transient entities (both with `null` id) are only equal by reference identity (`===`). Use `is` checks (Kotlin's `instanceof`) rather than `javaClass ==` to correctly handle Hibernate proxies, which are subclasses of your entity.

When a natural/business key exists (ISBN, email, tax ID), prefer it for both `equals()` and `hashCode()` — this is the gold standard because the key is immutable and known before persist.

Never include lazy collections or parent references in `toString()`. Keep it to `"${javaClass.simpleName}(id=$id)"`.

### ID generation: SEQUENCE over IDENTITY

**`GenerationType.IDENTITY` disables JDBC batching entirely.** With IDENTITY, the database generates the ID via auto-increment after each INSERT, forcing Hibernate to execute each INSERT immediately to retrieve the generated ID for its persistence context. It cannot defer or batch statements.

**`GenerationType.SEQUENCE`** lets Hibernate pre-fetch blocks of IDs (controlled by `allocationSize`) before executing any INSERTs, enabling full JDBC batching. On Aurora PostgreSQL, use native sequences. On Aurora MySQL (which lacks native sequences), Hibernate falls back to a TABLE-based sequence simulator, which works but has row-level locking overhead. If you're on MySQL and need batching, add `rewriteBatchedStatements=true` to the JDBC URL — the MySQL driver rewrites individual INSERTs into multi-value statements at the driver level.

For distributed systems, **UUID** generation (particularly time-ordered UUID v7 via `@UuidGenerator(style = Style.TIME)` in Hibernate 6.5+) avoids database round-trips entirely and supports batching, at the cost of 16-byte keys and potential B-tree fragmentation with random UUIDs.

### Relationship modeling in Kotlin

Always use **`MutableList`** (not `List`) for `@OneToMany` collections. Hibernate wraps your collection with its own `PersistentBag`/`PersistentList` proxy; it needs to add and remove elements. Using Kotlin's immutable `List` interface would require reassigning the entire collection, breaking Hibernate's dirty-tracking.

For bidirectional associations, always synchronize both sides with helper methods (`addComment`/`removeComment` as shown above). The child's `@ManyToOne` must always explicitly declare `fetch = FetchType.LAZY` — **the JPA default for `@ManyToOne` is `EAGER`**, which is the most common source of unintentional N+1 queries.

## 2. Repository patterns that leverage Kotlin's type system

### Choosing the right repository interface

Spring Data 3.0 restructured the hierarchy. **`JpaRepository<T, ID>`** remains the default choice — it provides `flush()`, `saveAndFlush()`, `deleteInBatch()`, and returns `List<T>` instead of `Iterable<T>`, which is more Kotlin-friendly. Use the bare `Repository` marker interface when you want to expose only specific methods (e.g., a read-only repository). `ListCrudRepository` (new in 3.0) returns `List` instead of `Iterable` if you want CRUD without JPA-specific methods.

### Kotlin nullability and the death of Optional

Spring Data introspects Kotlin's compiled nullability metadata via `kotlin-reflect`. Declaring `fun findByEmail(email: String): User?` returns `null` when no result is found. Declaring `fun findByEmail(email: String): User` throws `EmptyResultDataAccessException` if nothing matches. **Never use `Optional<T>` in Kotlin repositories** — use `T?` instead. For `findById`, use the built-in extension `findByIdOrNull`:

```kotlin
import org.springframework.data.repository.findByIdOrNull

val user = userRepository.findByIdOrNull(id) ?: throw UserNotFoundException(id)
```

Enable **`-Xjsr305=strict`** in your Kotlin compiler options so that Spring's `@NonNullApi` annotations become compile-time errors rather than platform types:

```kotlin
tasks.withType<KotlinCompile> {
    compilerOptions {
        freeCompilerArgs.add("-Xjsr305=strict")
    }
}
```

### @Query: triple-quoted strings, never string templates

Use Kotlin's triple-quoted strings for readable multi-line JPQL. **Never use `$variable` string interpolation inside `@Query`** — this is compile-time interpolation that creates SQL injection vulnerabilities. Always use `@Param` named parameters:

```kotlin
@Query("""
    SELECT u FROM User u
    WHERE u.department = :dept
    AND u.active = true
    ORDER BY u.createdAt DESC
""")
fun findActiveByDepartment(@Param("dept") dept: String): List<User>
```

For modifying queries, annotate with `@Modifying(clearAutomatically = true, flushAutomatically = true)` — `clearAutomatically` evicts stale entities from the persistence context after the bulk update.

### Projections: interfaces for derived queries, data classes for JPQL

Interface-based projections are the safest approach in Kotlin. Spring Data proxies the interface, and each `val` property maps to an entity attribute:

```kotlin
interface UserSummary {
    val firstName: String
    val lastName: String
    val email: String
}

interface UserRepository : JpaRepository<User, Long> {
    fun findByDepartment(dept: String): List<UserSummary>
}
```

Data class projections work with explicit JPQL constructor expressions (`SELECT new com.example.UserDto(u.name, u.email) FROM User u`) but have known issues with derived queries. For native queries, use interface projections with column aliases matching the property names.

Dynamic projections accept a `Class<T>` parameter, letting callers choose the projection type at the call site.

### Building a Kotlin specification DSL

`JpaSpecificationExecutor` produces verbose code. Wrap it with extension functions on `KProperty1` for type-safe, composable specifications:

```kotlin
fun <T> KProperty1<T, String?>.like(pattern: String): Specification<T> =
    Specification { root, _, cb -> cb.like(root.get(this.name), pattern) }

fun <T, V : Comparable<V>> KProperty1<T, V?>.greaterThan(value: V): Specification<T> =
    Specification { root, _, cb -> cb.greaterThan(root.get(this.name), value) }

infix fun <T> Specification<T>.and(other: Specification<T>): Specification<T> =
    Specification.where(this)!!.and(other)

infix fun <T> Specification<T>.or(other: Specification<T>): Specification<T> =
    Specification.where(this)!!.or(other)

// Usage — reads like a DSL
val spec = User::name.like("%alice%") and User::age.greaterThan(18)
userRepository.findAll(spec)
```

For complex queries beyond what Specifications offer, consider **Kotlin JDSL** (by LINE Corp), which provides a full JPQL DSL with Kotlin type safety.

### Custom repository fragments

The fragment pattern lets you inject `EntityManager` directly while keeping the standard repository interface clean:

```kotlin
interface UserRepositoryCustom {
    fun searchUsers(criteria: SearchCriteria): List<User>
}

class UserRepositoryCustomImpl(
    @PersistenceContext private val entityManager: EntityManager
) : UserRepositoryCustom {
    override fun searchUsers(criteria: SearchCriteria): List<User> {
        // CriteriaBuilder logic with full EntityManager access
    }
}

interface UserRepository : JpaRepository<User, Long>, UserRepositoryCustom
```

The implementation class **must** be named `{FragmentInterface}Impl`. Spring auto-detects it by convention — no `@Component` needed.

### Slice vs Page: skip the count query

`Page<T>` triggers an additional `SELECT COUNT(*)` on every request. For large tables, this count query often dominates execution time. **Use `Slice<T>` when your UI uses infinite scroll or "load more"** — it fetches `pageSize + 1` records to determine `hasNext()` without a count query. Reserve `Page<T>` only when you need "Page X of Y" display.

## 3. Performance: from N+1 to connection pool saturation

### Detecting and preventing N+1 queries

The N+1 problem is the single most common Hibernate performance issue. It manifests when loading N parent entities triggers N additional queries to fetch associated children. Key triggers include `FetchType.EAGER` on `@ManyToOne` (the JPA default), lazy collections accessed in loops, and bidirectional `@OneToOne` without `@MapsId`.

**Prevention strategy 1: JOIN FETCH in JPQL** — the primary solution for entity queries:
```kotlin
@Query("SELECT p FROM Post p JOIN FETCH p.comments WHERE p.status = :status")
fun findWithCommentsByStatus(@Param("status") status: PostStatus): List<Post>
```

**Prevention strategy 2: `@EntityGraph`** — declarative, no JPQL needed:
```kotlin
@EntityGraph(attributePaths = ["customer", "items"])
fun findAllByStatus(status: OrderStatus): List<Order>
```

**Monitoring**: Enable `hibernate.generate_statistics=true` in development and use `datasource-proxy` or P6Spy (`com.github.gavlyukovskiy:p6spy-spring-boot-starter:1.9.0`) to log all SQL with execution times. For automated test assertions, use Hibernate's `Statistics.getPrepareStatementCount()` or the Hypersistence Utils `SQLStatementCountValidator`.

### Why EAGER is almost always wrong

| Annotation | JPA Default | Recommendation |
|---|---|---|
| `@ManyToOne` | **EAGER** | Change to LAZY |
| `@OneToOne` | **EAGER** | Change to LAZY + `@MapsId` |
| `@OneToMany` | LAZY | Keep LAZY |
| `@ManyToMany` | LAZY | Keep LAZY |

EAGER cannot be overridden at query time — once set, every query loads that association. JPQL and Criteria API queries ignore entity-level fetch plans and fire separate SELECTs for EAGER associations, causing N+1 even when you thought you were safe. **Set every association to `FetchType.LAZY` and selectively fetch with `JOIN FETCH` or `@EntityGraph` per use case.**

### Batch inserts: configuration that actually enables batching

```yaml
spring:
  jpa:
    properties:
      hibernate:
        jdbc:
          batch_size: 25
        order_inserts: true
        order_updates: true
        batch_versioned_data: true
        connection:
          provider_disables_autocommit: true
  datasource:
    hikari:
      auto-commit: false
```

`order_inserts` groups INSERTs by entity type so Hibernate can batch them (without ordering, interleaved entity types break batching). `provider_disables_autocommit` eliminates an unnecessary `SELECT @@session.autocommit` query on every connection acquisition. Set `allocationSize` on `@SequenceGenerator` equal to or a multiple of `batch_size` to minimize sequence-fetch round-trips.

For large batch imports (**thousands of rows**), use the flush-and-clear pattern to prevent the first-level cache from growing unboundedly:

```kotlin
@Transactional
fun batchInsert(products: List<Product>) {
    products.forEachIndexed { index, product ->
        entityManager.persist(product)
        if ((index + 1) % 25 == 0) {
            entityManager.flush()
            entityManager.clear()
        }
    }
}
```

For even higher throughput, Hibernate's `StatelessSession` bypasses the persistence context entirely — no dirty checking, no L1 cache, pure JDBC batching.

### @Transactional(readOnly = true) — three layers of optimization

This annotation triggers optimizations at **three distinct levels**:

**Hibernate level**: Sets `Session.setDefaultReadOnly(true)`, which skips hydrated state snapshot storage (saves memory) and sets flush mode to `MANUAL` (skips dirty checking, saves CPU).

**JDBC driver level**: Calls `connection.setReadOnly(true)`. PostgreSQL sends `SET TRANSACTION READ ONLY` to the server. MySQL uses it as a driver hint.

**Aurora level**: Using `AbstractRoutingDataSource`, you can route read-only transactions to the Aurora **reader endpoint** and read-write transactions to the writer endpoint, distributing load across replicas:

```kotlin
class TransactionRoutingDataSource : AbstractRoutingDataSource() {
    override fun determineCurrentLookupKey(): Any =
        if (TransactionSynchronizationManager.isCurrentTransactionReadOnly())
            DataSourceType.READ_ONLY
        else DataSourceType.READ_WRITE
}
```

Best practice: annotate service classes with `@Transactional(readOnly = true)` as the default, then override individual write methods with `@Transactional`.

### Second-level cache: when and how

The L2 cache is a cross-session shared cache for entities with high read-to-write ratios. Configure it with JCache + Caffeine for single-instance deployments or Redisson for distributed setups:

```kotlin
// Entity annotation
@Entity
@Cacheable
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
class Country(@Id val id: Long, val name: String, val code: String)
```

**Use L2 cache for**: reference data (countries, categories, feature flags), rarely-modified lookup tables. **Avoid for**: frequently updated entities, entities with complex associations, or data requiring real-time consistency. The query cache (separate from entity cache) is invalidated whenever any entity of the cached type is modified — for frequently-written types, it adds overhead without benefit.

### HikariCP: the metrics that predict outages

The pool sizing formula is `connections = (CPU_cores × 2) + effective_spindle_count`. For Aurora's SSD-backed storage, spindle count ≈ 1. A 4-core database instance needs roughly 10 connections. With multiple application instances, multiply accordingly but stay well below Aurora's `max_connections`.

Critical metrics to alert on:

- **`hikaricp.connections.pending` > 0 sustained** — threads waiting for connections; pool is too small or transactions hold connections too long
- **`hikaricp.connections.timeout` > 0** — pool exhausted; immediate investigation required
- **`hikaricp.connections.usage` p99 > 1s** — connections borrowed too long, possible missing `@Transactional` boundaries or long-running queries
- **`hikaricp.connections.active` > 80% of max** — approaching saturation

Set `max-lifetime` to **28 minutes** (below Aurora's connection timeout), enable `tcpKeepAlive=true` to prevent NAT/firewall idle termination, and **always set `spring.jpa.open-in-view=false`** — the Open Session in View anti-pattern holds database connections for the entire HTTP request duration, including view rendering.

## 4. Testing against real databases, not H2

### @DataJpaTest with Testcontainers, not embedded databases

`@DataJpaTest` auto-configures `DataSource`, `EntityManager`, `TestEntityManager`, all `@Entity` classes, and all `@Repository` interfaces — while excluding web layer, service beans, and security. By default it replaces your datasource with an embedded H2 database, which has behavioral differences from MySQL/PostgreSQL (JSON types, windowing functions, locking semantics). **Always test against the real database engine.**

```kotlin
@TestConfiguration(proxyBeanMethods = false)
class PostgresTestConfig {
    @Bean
    @ServiceConnection  // Spring Boot 3.1+ — auto-configures datasource properties
    fun postgresContainer(): PostgreSQLContainer<*> =
        PostgreSQLContainer(DockerImageName.parse("postgres:16-alpine"))
}

@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
@Import(PostgresTestConfig::class)
class UserRepositoryTest(
    @Autowired val userRepository: UserRepository,
    @Autowired val entityManager: TestEntityManager
) {
    @Test
    fun `should find user by email`() {
        entityManager.persistAndFlush(createUser(email = "alice@test.com"))
        entityManager.clear()

        val found = userRepository.findByEmail("alice@test.com")
        assertThat(found?.email).isEqualTo("alice@test.com")
    }
}
```

`@ServiceConnection` (Spring Boot 3.1+) replaces the verbose `@DynamicPropertySource` boilerplate. The Spring context cache handles container reuse across test classes that share the same context configuration.

### Kotlin fixture factories beat Java builders

Kotlin's default arguments and named parameters eliminate the need for Builder classes entirely:

```kotlin
fun createUser(
    email: String = "user-${UUID.randomUUID()}@test.com",
    name: String = "Test User",
    role: Role = Role.USER,
    active: Boolean = true
): User = User(email = email, name = name, role = role, active = active)

// Tests specify only what's relevant:
entityManager.persistAndFlush(createUser(role = Role.ADMIN, active = false))
```

For deeply nested object graphs, a Kotlin DSL builder adds clarity:

```kotlin
fun order(block: OrderBuilder.() -> Unit): Order = OrderBuilder().apply(block).build()

val testOrder = order {
    status = OrderStatus.SHIPPED
    item { product = "Laptop"; quantity = 1 }
    item { product = "Mouse"; quantity = 2 }
}
```

### The flush-before-assert pattern prevents false positives

`@DataJpaTest` wraps each test in a transaction that rolls back afterward. This creates three categories of masked bugs:

**Lazy loading works when it shouldn't**: The test transaction keeps the session open, so lazy collections resolve fine. In production without OSIV, the same code throws `LazyInitializationException`.

**Missing `save()` calls pass**: Hibernate's dirty checking auto-flushes changes at transaction end. A test that modifies an entity without calling `save()` will pass, but production code without `@Transactional` will lose the change.

**Constraint violations go undetected**: `repository.save()` doesn't immediately flush. The rollback fires before the SQL hits the database, so unique constraint violations never surface.

**Always call `entityManager.flush()` then `entityManager.clear()` before assertions.** Flush forces SQL to the database (triggering constraint checks), clear evicts the L1 cache (ensuring subsequent queries hit the database).

For integration tests that verify transactional boundaries, **disable `@Transactional`** and use manual cleanup:

```kotlin
@DataJpaTest
@Transactional(propagation = Propagation.NOT_SUPPORTED)
class TransactionBoundaryTest {
    @AfterEach fun cleanup() = orderRepository.deleteAll()

    @Test
    fun `lazy loading fails outside transaction — matches production`() {
        val order = orderRepository.save(createOrder())
        assertThrows<LazyInitializationException> { order.items.size }
    }
}
```

### Asserting query counts catches N+1 regressions

```kotlin
@Test
fun `findAllWithItems should use single query`() {
    repeat(10) { entityManager.persist(createOrder(items = listOf(createItem()))) }
    entityManager.flush()
    entityManager.clear()
    statistics.clear()

    val orders = orderRepository.findAllWithItems()
    orders.forEach { it.items.size }  // Force lazy init

    assertThat(statistics.prepareStatementCount).isEqualTo(1)
}
```

## 5. Kotlin internals that break JPA in subtle ways

### Final classes and the silent proxy failure

Kotlin classes are final by default — a deliberate design choice promoting composition. Hibernate's ByteBuddy runtime generates proxy **subclasses** for lazy `@ManyToOne`/`@OneToOne` associations. When a class is final, proxy generation silently fails: Hibernate logs a warning (`HHH000112: Getters of lazy classes cannot be final`) but continues with eager fetching. Your application starts, tests pass with small datasets, and production crawls under N+1 load. The `allOpen` plugin configuration described in Section 1 is not optional — it is structurally necessary for lazy loading.

### The no-arg constructor's invisible nulls

The synthetic constructor generated by `kotlin-noarg` leaves all fields at JVM zero-values (`null` for objects, `0` for primitives). It does **not** run `init` blocks or property initializers unless `invokeInitializers = true` is set. For `val` properties, Hibernate uses `Field.setAccessible(true)` to override the `final` modifier via reflection and write the value. This works today but is threatened by **JEP 500 (Prepare to Make Final Mean Final)** — a proposed JDK change that would restrict reflection from modifying final fields. IntelliJ IDEA 2026.1 shows warnings for `val` fields in JPA entities. **Prefer `var` for all JPA-managed properties** to future-proof against this change.

### Coroutines and JPA: fundamentally incompatible

Spring Data JPA **explicitly declined** support for `suspend` functions in repositories (GitHub issue #3598). JPA's `EntityManager` and transaction state are stored in `ThreadLocal`. When a coroutine suspends and resumes on a different thread, the transaction context is lost, causing `TransactionRequiredException`. Attempting to declare `suspend fun findById(): Entity?` or `fun findAll(): Flow<Entity>` results in runtime errors like `Reactive Repositories are not supported by JPA`.

The workaround is wrapping blocking JPA calls in `withContext(Dispatchers.IO)`, but this still blocks a thread and **transaction context does not propagate across dispatchers**. For genuine non-blocking persistence, **Spring Data R2DBC** provides native coroutine support via `CoroutineCrudRepository` with `suspend` functions and `Flow<T>` return types — but it requires an R2DBC driver and gives up JPA's entity lifecycle, dirty checking, and L1 cache.

### Value classes: mostly broken, avoid for IDs

Kotlin value classes cause **name mangling** on getter methods — a property `val memory: Memory` (where `Memory` is an inline class) generates `getMemory-pMnLG0k()` with a hash suffix. Hibernate 6.x has added unmangling logic that handles this for regular entity properties, so value classes work as column-mapped fields. However, **using value classes as `@Id` types in Spring Data repositories fails** with `InvalidDataAccessApiUsageException: Provided id of the wrong type`. Collection parameters in `@Query` (`WHERE x IN :values` with `List<ValueClass>`) also fail due to unboxing issues. The `@JvmName` annotation can work around some issues, but the overall story is fragile. **Use value classes for domain logic and DTOs, not for JPA entity fields or repository ID types.**

### Field access is the Kotlin default — and that's fine

When you annotate a Kotlin property with `@Id`, the annotation targets the **backing field** by default. This means Hibernate uses `AccessType.FIELD` — reading and writing fields directly via reflection, bypassing getters and setters. This is generally the preferred mode: it provides better encapsulation (custom getter/setter logic in your code isn't triggered by Hibernate) and works with both `var` and `val` properties. The one caveat: with field access, calling `proxy.getId()` on an uninitialized lazy proxy **initializes the entire proxy** (triggering a query), whereas property access returns the ID directly. If you frequently access IDs on lazy proxies, consider `@Access(AccessType.PROPERTY)` on the `@Id` field specifically.

## Conclusion: the decisions that matter most

Three configuration choices prevent the majority of Kotlin+JPA production incidents. First, the `allOpen` block for `@Entity`, `@MappedSuperclass`, and `@Embeddable` — without it, lazy loading silently degrades to eager fetching across your entire entity graph. Second, `FetchType.LAZY` on every `@ManyToOne` and `@OneToOne` — the JPA default of EAGER is the root cause of most N+1 problems. Third, `spring.jpa.open-in-view=false` — holding database connections through view rendering is the leading cause of connection pool exhaustion under load.

Beyond configuration, the architectural insight is that **Kotlin and JPA have fundamentally different object models**. Kotlin's immutable `val`, null safety, data classes, value classes, and coroutines all conflict with JPA's mutable, reflection-heavy, thread-bound design. The pragmatic path is to accept JPA's constraints inside the entity layer (`var` properties, nullable types, regular classes) while using Kotlin's strengths everywhere else — factory functions for test fixtures, extension functions for repository DSLs, sealed classes for domain logic, and coroutines for non-persistence I/O. The entity layer is a thin, pragmatic bridge, not the place to showcase Kotlin idioms.
