# Mastery-level observability learning plan for Kotlin Spring Boot on EKS

**Observability is the ability to understand a system's internal state from its external outputs — and for a mid-level backend engineer running Kotlin microservices on Kubernetes, mastering it is the single highest-leverage skill for owning code in production.** This plan covers everything from foundational concepts through vendor selection to Spring Boot-native instrumentation, with curated resources prioritizing Spring Boot 3.x content as of early 2026. The plan is structured as three progressive areas: understand why observability matters, learn the tools landscape, then master Spring Boot's built-in capabilities.

---

## Area 1: Observability fundamentals and what they unlock in production

### Observability is not monitoring with a new name

Monitoring answers "is the system working?" by tracking predefined metrics against known thresholds. Observability answers **"why is the system broken?"** by enabling arbitrary questions about system behavior using high-cardinality, high-dimensionality data. The term comes from control theory: a system is "observable" if its internal states can be inferred from its outputs alone.

The distinction matters because distributed microservices break monitoring's assumptions. In a monolith, you could attach a debugger. In a mesh of Kotlin services on EKS, complexity lives in the **interactions between services**, not within any single component. Monitoring catches known-unknowns (CPU > 80%); observability reveals unknown-unknowns (why p99 latency spiked for users in Seoul hitting API v3 through a specific gateway pod).

Charity Majors, Honeycomb co-founder and the field's most influential voice, argues the "three pillars" framing (metrics, logs, traces) is misleading — those are data types, not observability itself. True observability is the **capability** to understand system state, regardless of data format. She distinguishes Observability 1.0 (separate tools for each signal type, limited correlation) from Observability 2.0 (unified columnar storage for wide structured events, where costs align with traffic rather than cardinality).

### The three pillars and the signals beyond them

**Metrics** are quantitative time-series measurements — request rate, CPU usage, error counts. Best for dashboards, trend analysis, and alerting on known thresholds. **Logs** are discrete timestamped event records — application errors, audit trails. Best for forensic investigation. **Traces** follow a single request's path through multiple services, capturing latency at each hop as spans. Best for understanding request flow in microservices.

Beyond the traditional three, production observability increasingly demands **continuous profiling** (stack traces and resource allocation at the code level, now adopted by OpenTelemetry as the emerging fourth pillar), **error tracking** (grouping and prioritizing exceptions with stack traces and user context — Sentry's specialty), **synthetic monitoring** (simulated user interactions from distributed locations to detect issues proactively), and **Real User Monitoring (RUM)** capturing actual user experience data.

These signals create the most value when **correlated**. A typical debugging workflow: an SLO alert fires → metrics dashboards identify the affected service → traces reveal which downstream call is slow → logs show the error details → profiles pinpoint the specific function consuming excessive CPU → RUM confirms real user impact. OpenTelemetry, the CNCF standard used by **76% of companies** (Grafana Labs 2025 Survey), unifies collection of all these signals.

### What observability enables and what its absence costs

Observability directly reduces **Mean Time to Detection (MTTD)** and **Mean Time to Resolution (MTTR)**. High-impact outages carry a median cost of **$2 million per hour** according to New Relic's 2025 report. The specific capabilities it unlocks include debugging latency by tracing slow requests across services, root cause analysis by slicing high-cardinality data by any dimension (user ID, region, deployment SHA), capacity planning through historical utilization trends, and incident response through correlated signals.

**SLOs, SLIs, and error budgets** are the governance framework that ties observability to business outcomes. An SLI (Service Level Indicator) is a quantitative metric like "percentage of requests returning 2xx within 200ms." An SLO (Service Level Objective) is the target — "99.9% over a 28-day rolling window." The error budget is the inverse: **0.1% failure tolerance, roughly 43 minutes of downtime per month**. When budget remains, teams ship features; when it's consumed, releases freeze. This bridges the tension between velocity and reliability.

Three case studies illustrate the cost of observability gaps:

- **Meta's October 2021 outage** (6 hours): A routine maintenance command disconnected all backbone routers. Facebook's internal tools depended on the same failing infrastructure — engineers couldn't access monitoring, communication tools, or even badge-controlled data center doors. Cost: **$60 million in lost revenue**, $6 billion drop in Zuckerberg's net worth.
- **CrowdStrike's July 2024 incident**: A faulty kernel update crashed **8.5 million Windows devices** globally. Lack of canary deployment and update pipeline monitoring allowed a content validation bug to ship worldwide simultaneously. Total Fortune 500 losses: **$5.4 billion**.
- **Cloudflare's November 2025 outage** (6 hours, ~20% of internet affected): A ClickHouse permissions change caused duplicate configuration rows, crashing proxy code. The team initially **misdiagnosed it as a DDoS attack** — better configuration pipeline observability would have caught the duplicate rows immediately.

### Instrumentation frameworks: Golden Signals, RED, and USE

The **Four Golden Signals** (Google SRE) form the foundation: latency, traffic, errors, and saturation. The **RED Method** (Tom Wilkie) focuses on request-driven microservices: Rate, Errors, Duration. The **USE Method** (Brendan Gregg) focuses on infrastructure resources: Utilization, Saturation, Errors. Best practice is RED for every microservice, USE for every infrastructure resource, with Golden Signals as the executive summary.

**What to instrument**: every service boundary (ingress/egress) with rate, latency, and error rate; database queries with duration; external dependency calls; queue depths and consumer lag; business-critical transactions (checkout, payment, auth); deployment metadata as trace attributes.

**What NOT to instrument**: don't try to monitor everything — you'll drown in noise. Avoid high-cardinality labels in traditional metrics systems (they'll explode costs). Minimize paging alerts to critical signals only — alert fatigue kills incident response. Don't create static dashboards for every metric; prefer interactive, queryable data.

### Observability maturity: four levels of organizational progression

Organizations progress through distinct maturity stages. **Level 1 (Reactive Monitoring)**: basic health checks, threshold alerts, siloed tools, manual root cause analysis. **Level 2 (Descriptive Observability)**: centralized logging, distributed tracing, correlated metrics across services via OpenTelemetry. **Level 3 (Predictive/Diagnostic)**: topology-aware correlation, anomaly detection, SLO-based alerting, predictive analytics. **Level 4 (Intelligent/AIOps)**: AI-driven root cause analysis, automated remediation, dynamic dashboards, predictive failure detection. The Grafana Labs 2025 survey found that organizations where the **C-suite views observability as business-critical** progress significantly faster.

### Key technical concepts to master

**Cardinality** — the number of unique values in a data field — is the most important cost and performance consideration. High-cardinality data (user IDs, request IDs) is the most valuable for debugging but most expensive to store. Moving from monolith to microservices can increase unique time series from 150,000 to **150 million**. Charity Majors notes: "World-class observability teams spend the majority of their time governing cardinality."

**Sampling strategies** are essential at scale. Head-based sampling decides at trace creation (simple but misses interesting traces randomly). Tail-based sampling decides after trace completion, keeping 100% of errors and high-latency traces — requires an OTel Collector to buffer spans. Best practice: head sampling for volume reduction, tail sampling at the collector for intelligent selection, with dynamic/adaptive rates per key.

**Observability-Driven Development (ODD)** treats observability as a design-time concern: dashboards and SLOs version-controlled as code, instrumentation tested in CI, developers verifying production telemetry after every deployment. Forrester (November 2025) described it as "a shift where real-time telemetry informs the SDLC, allowing decisions before firefighting is required."

## Area 2: The vendor and tools ecosystem for Spring Boot on EKS

### OpenTelemetry is the non-negotiable foundation

Regardless of which backend you choose, **OpenTelemetry should be your instrumentation layer**. It's the CNCF's second-most-active project after Kubernetes, providing a vendor-neutral framework for collecting metrics, traces, and logs via the OTLP protocol. The value proposition is simple: instrument once, export to any backend, switch backends by changing a config file.

For Kotlin Spring Boot, three integration paths exist. The **OTel Java Agent** (`opentelemetry-javaagent.jar`) provides zero-code auto-instrumentation for 150+ libraries (Spring MVC, WebFlux, JDBC, Kafka, gRPC) via `-javaagent` JVM flag. **Spring Boot's own starter** (new in 4.0, backport-compatible with 3.x) uses Micrometer as the observability API and exports via OTLP — this is the recommended path for new projects. The **community OTel Spring Boot Starter** offers a middle ground.

The **OTel Collector** is deployed on Kubernetes as a DaemonSet or sidecar, running a pipeline of receivers → processors → exporters. It can fan out to multiple backends simultaneously — sending traces to Jaeger AND X-Ray, metrics to Prometheus AND CloudWatch. This architecture decouples applications from backends entirely.

Spring Cloud Sleuth is **deprecated** as of Spring Boot 3. Its functionality migrated to **Micrometer Tracing** (`micrometer-tracing-bridge-otel` for OTel or `micrometer-tracing-bridge-brave` for Zipkin). New projects should use Micrometer Tracing + OTel directly.

### Commercial vendors: choosing the right combination

**Sentry** is the industry leader in error tracking and the most developer-friendly tool for debugging. Its Spring Boot integration via `sentry-spring-boot-starter-jakarta` (for Spring Boot 3+) auto-configures error capture, JDBC tracing via P6Spy, and logging integration through `sentry-logback`. Unique strengths include best-in-class error grouping, code-level stack traces with source context, breadcrumbs, and the new Seer AI debugging assistant. Pricing starts free (1 user), Team at **$29/month**, Business at **$80/month**, based on event volume. Limitation: not a full-stack platform — lacks infrastructure monitoring and log aggregation at scale. Best used alongside a broader APM tool.

**Datadog** offers the most comprehensive single-pane-of-glass platform: APM, infrastructure, logs, RUM, synthetics, profiling, security, with **900+ integrations**. The Java agent (`dd-java-agent.jar`) auto-instruments Spring Boot, Kotlin, Hibernate, Kafka, and more. On Kubernetes, deploy the Datadog Agent as a DaemonSet (one per node, not per pod — this distinction matters enormously for billing). Pricing is per-host: **$15/host/month** for infrastructure, **$31–$40/host/month** for APM (requires infrastructure plan), plus per-GB log ingestion and per-metric custom metrics charges. The "tag explosion" problem in Kubernetes can multiply custom metrics costs exponentially. Typical mid-size spend: **$50K–$150K/year**. Best for organizations that want everything in one platform and have the budget.

**New Relic** deserves special attention for Kotlin Spring Boot on EKS because the Java agent explicitly supports **Kotlin Coroutines v1.4+**, there are no per-host fees (critical for autoscaling EKS), and the free tier is genuinely generous: **100 GB/month data ingest, 1 full platform user, all 50+ capabilities, perpetual**. Standard pricing is consumption-based at $0.40–$0.60/GB beyond the free tier, with per-user fees ranging from $99/month (Standard) to $349/month (Pro). The steep Standard→Pro pricing cliff for teams >5 users is the main gotcha.

**Dynatrace** offers the most automated platform through **Davis AI** (combining causal, predictive, and generative AI for automatic root cause analysis). OneAgent installs once per host and auto-discovers everything — zero configuration. One customer reported **90% MTTI reduction**. Pricing is complex: $0.08/hour per 8 GiB host (~$58/host/month) for full-stack, $1.40–$3.60/month per container for Kubernetes. Best for large enterprises where MTTR reduction justifies the premium price.

**Honeycomb** is purpose-built for debugging complex distributed systems. Its signature **BubbleUp** feature automatically identifies which dimensions differ between problematic and baseline requests — genuinely unique among all vendors. Built on a columnar store with no cardinality limits. Fully OTel-native (no proprietary agent). Free tier includes ~20M events/month; Pro starts at ~**$130/month**. Named a Gartner Visionary in 2025. Best for SRE-mature teams that value deep debugging over dashboards.

**Splunk** offers **100% trace retention** (no sampling) — unique in the industry — combined with the strongest log management and SIEM capabilities. Its APM is OTel-native. Pricing starts at $60/host/month for APM + infrastructure. Best for regulated industries (fintech, healthcare) where compliance logging is mandatory.

**Elastic Observability** (ELK Stack) can be self-hosted for complete data control, combining observability with security (SIEM) in one platform. The APM Java agent auto-instruments Spring Boot. Self-hosted is free (open-source core); Elastic Cloud uses consumption-based pricing. Best for teams already using ELK for logs or requiring on-premises deployment.

**PagerDuty** remains the incident management standard with 700+ integrations, ML-powered alert grouping, and sophisticated escalation policies. Free for up to 5 users; Professional at $21/user/month. Note: **OpsGenie stopped new sales in June 2025** (shutdown April 2027) — Atlassian is migrating users to Jira Service Management.

For uptime monitoring, **Better Stack** (formerly Better Uptime) offers the most modern integrated platform combining monitoring, incident management, logs, and status pages. **UptimeRobot** provides the best free tier (50 monitors, 5-minute checks). **Pingdom** adds RUM capabilities but lacks a free tier.

### Open-source stack: Prometheus, Grafana, and friends

The **Prometheus + Grafana** combination is the open-source standard for Kubernetes metrics. Spring Boot integration requires just two dependencies (`spring-boot-starter-actuator` + `micrometer-registry-prometheus`) and one config line to expose `/actuator/prometheus`. Deploy on Kubernetes via the **kube-prometheus-stack Helm chart**, which includes Prometheus Operator, Alertmanager, Grafana, node-exporter, kube-state-metrics, and pre-configured dashboards. Uses ServiceMonitor CRDs for auto-discovery of scrape targets.

**Grafana Cloud** manages the full LGTM stack: Mimir (metrics), Loki (logs), Tempo (traces), Pyroscope (profiling). Its free tier includes **10K metric series, 50GB logs, 50GB traces** with 13-month metric retention — genuinely useful for small deployments. Pro starts at $19/month. The platform offers Adaptive Metrics and Adaptive Logs to control costs automatically.

**Jaeger** (CNCF graduated) is the production-grade distributed tracing platform built by Uber. Deploy via Jaeger Operator on Kubernetes with OTLP endpoints. Strong adaptive sampling and excellent trace comparison UI. **Zipkin** is simpler to deploy (single Docker container) but losing mindshare as the ecosystem moves to OTel-native solutions. For new projects, Jaeger or Grafana Tempo are preferred.

**SigNoz** is the strongest open-source alternative to Datadog — an OTel-native platform providing metrics, traces, and logs in a single UI, backed by ClickHouse for fast queries. No per-user or per-host fees; cloud pricing at $0.30/GB for logs and traces. Its community has grown to **24K+ GitHub stars**. Best when you want one platform without assembling Prometheus + Loki + Tempo + Grafana.

### AWS native observability on EKS

The AWS stack consists of **CloudWatch** (metrics, logs, alarms, Container Insights), **X-Ray** (distributed tracing), and **ADOT** (AWS Distro for OpenTelemetry). Install the CloudWatch Observability EKS add-on to deploy Fluent Bit for log collection and the CloudWatch agent for infrastructure metrics. Container Insights auto-discovers EKS clusters and provides cluster/node/pod/namespace-level metrics.

**Critical note**: X-Ray SDKs are entering **maintenance mode on February 25, 2026** — AWS recommends migrating to ADOT. Use the ADOT Java agent (OTel-based) with the ADOT Collector configured for X-Ray and CloudWatch exporters. AWS **Application Signals** (`OTEL_AWS_APPLICATION_SIGNALS_ENABLED=true`) automatically creates APM dashboards showing request volume, latency percentiles, and error rates.

Estimated cost for a small EKS cluster (10 nodes, 20 pods) with moderate traffic: **~$125/month**. CloudWatch's complex billing (15+ dimensions) and Container Insights' metric generation can cause surprises. Use ADOT Collector filter processors to control cardinality.

### Recommended stacks by budget and maturity

**Budget-conscious startup**: Sentry (free) + New Relic (free tier, 100GB) + UptimeRobot (free) + PagerDuty (free, 5 users) — **$0–$110/month**.

**Mid-size team**: Sentry (Team, $29/month) + Grafana Cloud (Pro) or New Relic (Standard) + PagerDuty (Professional) + Better Stack (uptime) — **$500–$3,000/month**.

**Enterprise**: Dynatrace (full-stack with Davis AI) + Sentry (Business) + PagerDuty (Enterprise) — **$5,000+/month**.

**Open-source maximalist**: SigNoz (self-hosted) or Prometheus + Loki + Tempo + Grafana + Sentry (self-hosted) — infrastructure costs only.

## Area 3: Spring Boot's native observability capabilities

### Spring Boot Actuator is your observability control plane

Actuator provides production-ready monitoring endpoints. In Spring Boot 3.x, **only `/actuator/health` is exposed over HTTP by default** — a security improvement over 2.x. Key endpoints to understand:

**Core monitoring**: `/health` (aggregated health status for K8s probes), `/info` (build metadata, git commit), `/metrics` (Micrometer metrics — JVM, HTTP, DB, cache, custom), `/prometheus` (Prometheus exposition format, requires `micrometer-registry-prometheus`).

**Diagnostics**: `/threaddump` (JSON thread dump for debugging deadlocks), `/heapdump` (binary heap dump for memory leak analysis — **critical security risk**), `/loggers` (view and change log levels at runtime via GET/POST without restart), `/scheduledtasks` (all @Scheduled tasks with execution timing).

**Configuration**: `/env` (all configuration properties — can expose secrets), `/beans` (all Spring beans with dependencies), `/conditions` (auto-configuration evaluation), `/mappings` (all @RequestMapping paths), `/configprops` (all @ConfigurationProperties values).

**Specialized**: `/startup` (application startup timing per bean, requires `BufferingApplicationStartup`), `/caches` (cache managers with eviction support), `/flyway` and `/liquibase` (migration state), `/sbom` (Software Bill of Materials, new in 3.x).

### Actuator security is non-negotiable after the Volkswagen breach

In December 2024, researchers found an unprotected `/actuator/heapdump` endpoint in a Volkswagen telematics service. The heap dump contained **plaintext AWS credentials**, leading to the download of **9 terabytes of GPS data** from hundreds of thousands of vehicles. This was a misconfiguration, not a zero-day.

Classify endpoints by risk: **Low** (health without details — safe for public probes), **Medium** (info, metrics, prometheus — internal network or authenticated), **High** (env, loggers, configprops, beans, threaddump — role-based only), **Critical** (heapdump, shutdown — **never expose**).

The recommended production configuration uses a **separate management port** accessible only from the monitoring namespace:

```yaml
management:
  server:
    port: 8081
  endpoints:
    web:
      exposure:
        include: health,info,prometheus
        exclude: env,heapdump,beans,loggers,threaddump,shutdown,configprops
  endpoint:
    health:
      show-details: when-authorized
      probes:
        enabled: true
```

Pair this with a Kubernetes NetworkPolicy restricting port 8081 access to the monitoring namespace. For Spring Security, permit `/actuator/health` unauthenticated (for K8s probes), require the `ACTUATOR` role for everything else.

### Health groups and Kubernetes probes work together

Spring Boot 3's health groups map directly to Kubernetes probes. The **critical design rule**: liveness probes should NEVER include external dependencies. If your database goes down and the liveness probe fails, Kubernetes restarts the pod — which won't fix the database and creates cascading failures. External dependencies belong in readiness only.

```yaml
management:
  endpoint:
    health:
      group:
        liveness:
          include: livenessState    # Only JVM liveness — no DB, Redis, Kafka
        readiness:
          include: readinessState,db,redis,paymentGateway
```

Custom health indicators in Kotlin implement `HealthIndicator` (blocking) or `ReactiveHealthIndicator` (WebFlux). Use `@ConditionalOnEnabledHealthIndicator` for conditional activation. The startup probe (using `failureThreshold: 30` × `periodSeconds: 5s` = 150s max) gives the JVM time to start, after which liveness and readiness probes take over.

Graceful shutdown requires `server.shutdown=graceful` and `spring.lifecycle.timeout-per-shutdown-phase=30s`. The sequence: Kubernetes sends SIGTERM → readiness returns 503 → K8s removes pod from Service endpoints → Spring finishes in-flight requests → clean shutdown.

### Micrometer and the Observation API are Spring Boot 3's biggest observability leap

The **Observation API** is the most important architectural change in Spring Boot 3.x. Previously, instrumenting for metrics AND tracing required two separate APIs (Micrometer for metrics, Sleuth for tracing). Now you **observe once** and get metrics, traces, and log correlation automatically:

```
Your Code → Observation API → ObservationHandler(Tracing)  → OpenTelemetry
                             → ObservationHandler(Metrics)  → Micrometer Registry
                             → ObservationHandler(Logging)   → MDC correlation
```

The `@Observed` annotation (requires `spring-boot-starter-aop`) automatically creates a timer metric AND a trace span for any method. The programmatic API offers finer control with `Observation.createNotStarted()`, distinguishing between **low-cardinality** key-values (appear in both metrics and traces — e.g., `method`, `status`) and **high-cardinality** key-values (traces only — e.g., `userId`, `orderId`). This distinction directly addresses the cardinality cost problem.

Spring Boot auto-configures metrics for JVM (heap, GC, threads, buffers), HTTP server requests (tagged by URI, method, status, outcome), **HikariCP connection pools** (active, idle, pending, timeout, usage, acquisition time), caches, Kafka consumers/producers, Hibernate queries, and application startup timing. No code required — just add `spring-boot-starter-actuator`.

The HikariCP metrics deserve special attention: `hikaricp.connections.pending` (threads waiting for a connection) and `hikaricp.connections.timeout` (acquisition timeouts) are **early warning indicators of database saturation** that should trigger alerts immediately when sustained above zero.

### Custom metrics and endpoints in Kotlin

Custom metrics use Micrometer's API — `Counter`, `Gauge`, `Timer`, `DistributionSummary` — registered through the `MeterRegistry`. Enable percentile histograms for meaningful latency tracking:

```yaml
management:
  metrics:
    distribution:
      percentiles-histogram:
        http.server.requests: true
      percentiles:
        http.server.requests: 0.5,0.95,0.99
```

Custom Actuator endpoints use `@Endpoint(id = "myendpoint")` with `@ReadOperation` (GET), `@WriteOperation` (POST), and `@DeleteOperation` (DELETE). The `@Selector` annotation creates path variables. Web-only endpoints use `@WebEndpoint`. All custom endpoints must be explicitly included in the exposure configuration.

**Spring Boot Admin** (codecentric, version 3.5.x) provides a web UI for monitoring multiple Spring Boot applications. It discovers services via direct client registration or Spring Cloud/Kubernetes service discovery, showing health status, metrics, environment, thread dumps, and dynamic log level management across all instances. It supports notifications via Slack, PagerDuty, Email, and Teams.

## Conclusion: a practical learning sequence

The most efficient path through this material follows a specific sequence. **Week 1–2**: Read Cindy Sridharan's free ebook and the Observability Engineering book chapters on concepts, SLOs, and the maturity model. Understand why before how. **Week 3–4**: Clone the `teppana88/spring-opentelemetry` Kotlin repo and the `blueswen/spring-boot-observability` repo. Run them locally with Docker Compose. Experiment with Actuator endpoints, Micrometer metrics, and the Prometheus/Grafana stack hands-on. **Week 5–6**: Study the Spring team's OTel blog post and Baeldung's Spring Boot observability guide. Implement the `@Observed` annotation and custom metrics in your own services. Configure health groups for Kubernetes probes. **Week 7–8**: Evaluate vendors against your team's needs using the stack recommendations above. Set up a trial of your chosen combination (e.g., Sentry free + Grafana Cloud free + OTel Collector on your EKS cluster).

The single most impactful insight across all this research: **OpenTelemetry + Spring Boot 3's Observation API + proper health group configuration for Kubernetes probes** forms the non-negotiable foundation. Every vendor choice and architectural decision builds on these three elements. Master them first, then layer on vendor-specific capabilities based on your team's budget, scale, and maturity level.
